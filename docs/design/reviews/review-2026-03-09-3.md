### Unique Findings (Single Reviewer)

**Connector remains registered after validation failure** (kimi, medium). `registry.register()` executes unconditionally before `validate()` in `server.py:2716-2728`. If validation fails, the connector stays in the registry with "error" status but is still returned by `registry.get()`. No other reviewer caught this. It's a real logic error — registration should happen after validation, or the connector should be unregistered on failure.

**API contract drift from spec** (gpt-5.3, medium). The request model uses `connector_names` + `filters` instead of `connectors` + `artifact_type`, and the response uses `content` instead of `content_preview` per `connector_models.py:63-76`. No other reviewer flagged this. Functionally harmless but creates spec divergence that could bite during future integrations.

**`query_federated` timeout not caller-configurable via MCP** (minimax, minor). The 30-second default is hardcoded at the endpoint level. Callers can't override it. Minor ergonomics issue.

**Registry doesn't enforce `connector_name` attribution** (gpt-5.3, low). Built-in connectors set it correctly, but custom connectors could omit it silently.

### New Findings

**Medium — Unregistered connector on validation failure**

At `server.py:2718`, when validation fails, the connector status is set to "error" but the connector instance is **already registered** in the registry (line 2718 happens inside the try block after line 2716's `registry.register(instance)`). The registration at line 2716 is unconditional. This means:
- A connector that failed validation remains in the registry
- Future calls to `registry.get(name)` will return a connector that may be misconfigured
- The connector stays registered until server restart

The fix: only register after successful validation, or unregister on failure. Currently:
```python
registry.register(instance)  # Always happens
try:
    await instance.validate()  # May fail
except ...
    status = "error"
    # instance remains registered!
```

**Low — `sweep_connector_index` config parsing incomplete**

At `embedding.py:284`, the config parsing handles JSON strings but the logic is slightly convoluted:
```python
config = row["config"] if isinstance(row["config"], dict) else json.loads(row["config"])
```
This works, but if `row["config"]` is a JSON string representing a non-dict (e.g., `"null"`, `"[]"`), it will crash later when accessing `config.get("root_path")`. Add a type check:
```python
config = row["config"] if isinstance(row["config"], dict) else json.loads(row["config"])
if not isinstance(config, dict):
    config = {}  # Defensive: handle corrupted config
```

**Low — Missing `embedding IS NOT NULL` guard (carried from both prior reviews)**

At `filesystem_connector.py:286`, the query includes the guard (`AND embedding IS NOT NULL`), so this is actually **fixed**. The prior reviews noted it as missing but current code shows it's present. The review-2026-03-09.md finding can be marked resolved.
