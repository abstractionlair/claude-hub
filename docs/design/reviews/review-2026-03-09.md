### 1. BaseConnector Interface Design

**Does it support both "query in place" and "index and cache" strategies?**

Yes. The design is sound. `query()` is abstract and required — every connector must support search. `index()` is optional with a `NotImplementedError` default — connectors that don't need caching (like ArtifactConnector, which embeds on write) simply don't override it. This cleanly separates the two strategies. `write()` follows the same optional pattern.

One design consideration: the `validate()` method returns `bool` but the docstring says "raise ConnectorError with details if not." This is internally consistent (return True on success, raise on failure, never return False), but the return type annotation `-> bool` is misleading — `-> True` or `-> Literal[True]` would communicate the contract more precisely. The current annotation suggests `False` is a valid return, but no caller should ever receive it. **Low severity** — cosmetic, not a bug.

### 2. ArtifactConnector Delegation

**Finding (Medium): `list()` ignores the `path` parameter.** At `artifact_connector.py:91-120`, the `path` parameter is accepted but never forwarded to `artifact_store.list_artifacts()`. The spec says `list(path)` means "sub-path to list within (None = root)." For the artifact store, `path` could reasonably map to `artifact_type` filtering, but currently it's silently dropped. This means callers who pass `path` expecting filtering get back unfiltered results with no indication.

**Finding (Low): `list()` also ignores `artifact_type` from filters.** Unlike `query()` which extracts `filters.get("artifact_type")`, the `list()` method doesn't accept or forward filter parameters at all. `artifact_store.list_artifacts` supports `artifact_type` and `tags` filtering, but the connector wrapper doesn't expose this.

**Finding (Low): `query()` doesn't forward `date_from`, `date_to`, `include_archived`, or `confidence` filters.** `artifact_store.search_artifacts` accepts these parameters, but the connector's `filters` dict only extracts `artifact_type` and `tags`. The connector interface's generic `filters: dict` was designed to be connector-specific, so this isn't wrong per se, but it's an undocumented limitation. Users of the federated query who want date-range filtering through the artifact connector won't get it.

**Delegation is otherwise correct.** The `query()`, `list()`, and `get()` methods call the right `artifact_store` functions with the right pool parameter. The result mapping from dicts to `ConnectorResult`/`ConnectorItem` is faithful. The `get()` method correctly raises `KeyError` when `get_artifact` returns `None`.

### 3. ConnectorRegistry Federated Query Scalability

**Will it scale to 5-10 connectors?**

Yes, comfortably. The implementation at `registry.py:107-126` uses `asyncio.gather(*tasks, return_exceptions=True)` to fan out queries in parallel. With 5-10 connectors, all queries execute concurrently. The merge is a simple sort of the combined results list — at the expected result count (10-50 per connector × 10 connectors = 100-500 items), the sort is negligible.

**Finding (Low): No per-connector timeout.** If one connector hangs (e.g., a slow filesystem scan or a network-dependent future connector), the entire `federated_query` blocks waiting for `asyncio.gather` to complete. With 5-10 connectors, one slow connector degrades the experience for all queries. The spec mentions timeout handling in the review engine but doesn't specify connector-level timeouts. Consider wrapping each `connector.query()` call with `asyncio.wait_for(connector.query(...), timeout=30)` to isolate slow connectors. The error handling already catches exceptions, so a `TimeoutError` would be handled gracefully.

**Finding (Low): `connector_name` not set on results from `federated_query`.** At `registry.py:123`, `merged.extend(outcome)` adds results as-is. The `connector_name` field is only set if the individual connector sets it in its own `query()` method. The spec's `federated_query` (Section 5.2) explicitly sets `r.connector_name = connector.name` for each result. `ArtifactConnector.query()` sets it at line 87, and `FilesystemConnector.query()` sets it at line 252. So the current implementations are fine, but the registry doesn't enforce it — a future connector that forgets to set `connector_name` would return results with empty connector attribution. The spec intended the registry to set it as a post-processing step. This is a minor divergence from spec.

### 4. FilesystemConnector Index/Query/Dedup Logic

**Dedup is correct.** The `index()` method at `filesystem_connector.py:107-219` checks for existing entries by `(connector_id, source_path)`, compares `content_hash`, and either skips (unchanged), updates (changed), or inserts (new). The dedup is backed by the `UNIQUE(connector_id, source_path)` constraint in the migration. SHA-256 hashing uses the same algorithm as the artifact store.

**Finding (Medium): `index()` embeds full file content but stores only 500-char preview.** At line 134, `content_preview = content[:500]`. At line 190, `embedding = await generate_embedding(content)` — this embeds the *full* file content. But later, the `sweep_connector_index` function in `embedding.py:246-316` embeds `content_preview` (the 500-char truncation). This means:
- First index: embedding is generated from full content (correct).
- Retry sweep: embedding is generated from 500-char preview (incorrect — degraded quality).

This mismatch means if the initial embedding fails and the retry loop picks it up, the resulting embedding will be lower quality. The `connector_index` table doesn't store the full content — only `content_preview`. The retry loop has no way to read the full file because it doesn't know the `root_path`. This is a genuine data integrity issue with the fallback path.

**Finding (Medium): Path traversal in `get()`.** At `filesystem_connector.py:294`, `file_path = self._root_path / item_id`. If `item_id` contains `../`, an attacker (or buggy caller) could read files outside the root directory. For example, `get("../../etc/passwd")` would resolve to a path outside `root_path`. The `get()` method doesn't validate that the resolved path is within `root_path`. This should be:

```python
file_path = (self._root_path / item_id).resolve()
if not file_path.is_relative_to(self._root_path.resolve()):
    raise KeyError(f"Path escapes root: {item_id}")
```

Similarly, `index()` at line 115 does `scan_root = self._root_path / path if path else self._root_path` — the `path` parameter in `index()` could also traverse outside the root.

**Finding (Low): No `embedding IS NOT NULL` guard in the query SQL.** At `filesystem_connector.py:231-244`, the query filters on `embedding_status = 'complete'` but doesn't guard against `embedding IS NOT NULL`. The spec (Section 2.7 design notes) explicitly mentions this: "The `ae.embedding IS NOT NULL` guard prevents null vector distance calculations." If a row somehow has `status = 'complete'` but a NULL embedding (data corruption, manual DB edit), the `<=>` operator would fail or produce unexpected results. The `embedding_status = 'complete'` filter makes this unlikely but not impossible.

**Query logic is sound.** The vector similarity search at `filesystem_connector.py:231-244` correctly uses `1 - (embedding <=> $1::vector)` for cosine similarity, filters by `connector_id` and `embedding_status`, and orders by distance. The `generate_query_embedding` function (with `RETRIEVAL_QUERY` task type) is correctly used for search queries vs `generate_embedding` (`RETRIEVAL_DOCUMENT`) for indexing.

### 5. Migration Schema

**Finding (Low): HNSW index on potentially-NULL embedding column.** The migration at `005_connectors.sql:33-35` creates an HNSW index on `connector_index.embedding`. Rows inserted with `embedding_status = 'pending'` have NULL embeddings. pgvector's HNSW index handles NULL entries (they're excluded from the index), so this works, but it's worth noting that the index is only useful for the subset of rows where `embedding IS NOT NULL`.

**The schema matches the spec.** Tables, columns, constraints, and indexes all align with Section 2.5. The `IF NOT EXISTS` guards make the migration idempotent.

The models look correct for this flow, but the wiring doesn't exist yet — something to verify during integration.

**The models are clean and well-structured.** `QueryFederatedResponse` includes both `results` and `total` count. `ConnectorRegisterResponse` echoes back the registration details. Field descriptions and validation constraints (`ge=1, le=100`) are appropriate.
