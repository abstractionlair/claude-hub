### Critical findings

**1. `limit_price_cents` is accepted but ignored by `FlowModel.estimate()`** (flow_model.py:244). Calibration simulates virtual orders at five price points (flow_model.py:396) but stores observations under keys that omit price entirely (flow_model.py:402). The result: fill probability is invariant to the limit price at inference time. This means EVStrategy's joint (side, price, quantity) search (ev_strategy.py:256) isn't actually finding a price/fill frontier — the fill estimate is the same at every price step for a given quantity. Additionally, observation counts are inflated by the number of simulated prices (5x), which can push bins over `MIN_COMBINED`/`MIN_PER_OUTCOME` thresholds prematurely. No test checks price dependence.

This is a real finding. The spec envisions price-dependent CDFs but the implementation collapses price into a single bin. The practical impact depends on how much fill probability actually varies by price — if the CDF at 87c vs 95c is materially different, this is a significant calibration error. If not, it's "working by accident" but still a design gap.

**2. Replay `--flow-fills` path never fills orders** (replay.py:574). The FlowModel branch creates a Bernoulli fill schedule, but the fill-processing loop (replay.py:427) only consumes schedules when `use_tape_fills` is true, not `use_flow_model`. Orders placed via the FlowModel path stay pending forever and expire unfilled at settlement. No test covers the `replay(..., use_flow_model=True)` path.

This is clearly correct — the condition at line 427 checks `use_tape_fills`, not `use_tape_fills or use_flow_model`, even though both paths use `fill_schedule`. This is a straightforward integration bug.

### High severity

**3. `MarketView.fill_rates()` FlowModel fallback is a stub** (market_view.py:121-126). When FlowModel is the only backend, `fill_rates()` hardcodes `quantity=1, limit_price_cents=90, trailing_volume=0` regardless of the `relative_price` argument. This means legacy callers that expect price-sensitive fill rates get a constant value. The reviewer flags this as a backward-compatibility break.

This is accurate. The method signature takes `relative_price` but the FlowModel fallback ignores it entirely. Whether this matters depends on whether any code path actually calls `fill_rates()` when FlowModel is active — `_find_best_order` uses `fill_estimate()` instead, so this may only affect `_find_best_limit` (the non-FlowModel path). But if someone constructs a MarketView with FlowModel and then calls the legacy path, it's silently wrong.

### Low severity

**4. `compute_trailing_volume()` is O(n) per call in replay** (replay.py:477, flow_model.py:145). Each call rebuilds the timestamp list and sums counts, unlike calibration which uses prefix sums. On long tapes across many periods this adds up, but it's a performance issue, not a correctness one.

### What the review did NOT find (gaps)

The reviewer noted the opposing-flow direction logic (flow_model.py:119) is correct, and that test coverage for interpolation, threshold fallthrough, and tape-fill directionality is solid. The review was limited by a sparse checkout (no access to `fill_rate.py`, `event_rate.py`, `cost_model.py`, `strategy.py`, or the spec document), so it could not verify cross-module integration beyond what's visible in the target files.
