### High Severity

**Synthesis output not unwrapped from JSON (lines 1246-1247)**

```python
if result.returncode == 0 and result.stdout.strip():
    synthesis_prose = result.stdout.strip()
```

The synthesis model is invoked using `synth_model_config["invoke"]`, which for Claude includes `--output-format json`. The CLI will therefore return a JSON object like `{"session_id": "...", "result": "actual prose"}`. But the code assigns `result.stdout.strip()` directly to `synthesis_prose` without calling `_extract_session_and_text` to unwrap the JSON envelope.

This means `synthesis_prose` — which flows into the artifact store, the peer-disagreement prompt, and the grading prompt — is JSON-wrapped, not clean prose. The downstream models will receive something like `{"session_id":"abc","result":"Here are 4 model reviews..."}` instead of the actual synthesis text.

Every review invocation calls `_extract_session_and_text` at line 927, but the synthesis invocation at line 1246 does not. The fix is straightforward: apply the same extraction here.

This also means the synthesis model's session_id is silently discarded, which means the contradiction detection step (line 1434) starts a fresh session instead of resuming the synthesizer's context. That's a secondary loss — the synthesizer has to re-read the reviews to identify contradictions rather than building on its own work.

### Medium Severity

**Double call to `_check_and_synthesize` on failure paths (lines 900, 922, 1040)**

On the failure paths (non-zero exit code, empty output), the code explicitly calls `_check_and_synthesize` and then returns. But `return` from a `try` block still executes the `finally` clause, which calls `_check_and_synthesize` again at line 1040. Since the synthesis row is only inserted after synthesis completes (line 1352-1373, not at the start of `_check_and_synthesize`), both calls can pass the `already_synthesized` guard and each spawn a full synthesis run — subprocess invocations, grading, artifact storage — all doubled.

The success path is fine (no early return, only the `finally` call fires). The fix: remove the explicit calls at lines 900 and 922, since the `finally` block handles it.

**Race window in `_check_and_synthesize` — no placeholder row (lines 1060-1091)**

The function uses a transaction to check whether synthesis is needed, but does not insert a placeholder row. Two concurrent `_run_single_review` tasks finishing simultaneously could both pass the guard (`already_synthesized` is None for both) and spawn concurrent synthesis runs. The `UniqueViolationError` catch at line 1374 prevents duplicate DB rows, but two full synthesis subprocesses (plus grading passes) run to completion before one fails on insert. On a 4GB VPS with a semaphore limit of 3, this could compete with active reviews for resources.

Fix: insert a sentinel synthesis row (e.g., with empty data) inside the transaction at line 1061-1078, then update it after synthesis completes.

**Cross-grading cycle count may never exceed 20 for synthesizer-only entries (line 1626-1628)**

```python
cross_grade_count = await conn.fetchval(
    "SELECT COUNT(DISTINCT job_id) FROM review_grades WHERE grader_model IS NOT NULL"
) or 0
```

This counts all jobs that have *any* non-null grader_model. Once the system switches to synthesizer-only grading after 20 cycles, the grader_model is still non-null (it's the synthesizer's name). So `cross_grade_count` continues incrementing and `cross_grade_count % 5 == 0` triggers full cross-grading every 5 cycles. That part works.

However, the query doesn't distinguish between "full N×N cross-grading" and "synthesizer-only grading." If you wanted to only count full cross-grading cycles (to determine when the 20-cycle calibration period ends), this query overcounts. Cycle 1 through 20 are full cross-grading; but if cycle 21 is synthesizer-only, it still increments the counter, so cycle 22 is `22 % 5 != 0` → synthesizer-only, cycle 25 is full again, etc. The policy as coded works fine for the mod-5 case, but the "first 20" threshold was crossed during the period where all grading was full N×N, so calibration data is correct. Acceptable but worth noting the semantics.

### Low Severity

**Unnecessary temp directory creation in peer follow-up loop (lines 1532-1534, 1594-1596)**

Inside the per-position follow-up loop, a temp directory is created (`tempfile.mkdtemp(prefix="followup_")`) but never used — the follow-up prompt is passed via `input=followup_prompt` in `subprocess.run`, not via a temp file. The `tmp_dir` is created and cleaned up for no reason. Minor resource waste per contradiction position.

**Greedy regex in `_parse_grades_json` (line 1912)**

```python
match = re.search(r'\[.*\]', raw, re.DOTALL)
```

With `DOTALL`, `.*` is greedy and matches from the first `[` to the last `]` in the entire output. If a model emits explanatory text containing square brackets before the actual JSON array, the regex grabs too much and JSON parsing fails. A non-greedy match wouldn't help (it would match too little). A more robust approach would be to iterate through `[` positions and try parsing from each one.

**Sequential follow-up and grading could be parallelized**

`_peer_followup` processes each contradiction and each reviewer position sequentially (lines 1489-1598). `_grade_reviewers` runs each grader model sequentially (line 1672-1684). With 4 models and multiple contradictions, this serializes work that could run concurrently with `asyncio.gather`. Not a correctness issue, but on the 4GB VPS the wall-clock time adds up — each follow-up or grading call can take minutes.

**`.review-external` directory never cleaned up (line 386-398)**

The internalized-files directory is created in the project root but there's no cleanup after the job completes. Repeated reviews accumulate stale copies. The `finally` block in `_run_single_review` cleans up the per-review temp dir but not this project-level directory.

**`all_failed` logic in `check_review_status` (lines 546-561)**

```python
all_failed = True
...
if status not in ("failed",):
    all_failed = False
```

A review with status "pending" or "running" sets `all_failed = False`, but `done` doesn't increment for those statuses. So `all_failed and done == total` (line 567) can never be true when there are pending/running reviews — `done < total` takes precedence. The logic is correct but the `all_failed` variable is misleading; it really means "no non-failed review exists at all," not "all completed reviews failed." Not a bug, just confusing.

### Design Observations (not bugs)

**Peer follow-up results are only visible in `review_grades.peer_disagreement`, not in the synthesis artifact.** The synthesis prose is stored before peer follow-up runs (line 1274 calls `_peer_followup` after synthesis subprocess completes). Someone reading the synthesis artifact won't see whether contradictions were resolved. The resolution data flows only into the grading step.

**Self-grading in cross-grading mode.** When full cross-grading is active, each reviewer model grades all reviews including its own (lines 1643-1647). This is presumably intentional for calibration, but self-assessment is known to be unreliable — a model is likely to rate its own output favorably. No filtering on `grader_name == reviewed_model_name` exists.

**Codex JSONL format resilience.** The `_extract_session_and_text` codex parser (lines 1869-1889) silently skips non-JSON lines, which is appropriate for JSONL. It only captures text from `item.completed` events with `agent_message` type. If codex changes its event schema (e.g., different event types for final output), the parser would return no text and fall back to `raw_output`. The fallback is safe but would store unparsed JSONL as the review content.

**`review_target` column redundant with `review_type`** (line 1819). The comment "for now" suggests future differentiation. Currently every grade row stores the same value in both columns.
