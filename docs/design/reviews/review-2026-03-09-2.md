### New Findings

**Medium — Stale cleanup deletes entries outside the scan scope during scoped indexing.**

At `filesystem_connector.py:201-229`, the stale cleanup runs unconditionally. When `index(path="subdir")` is called, `files` only contains entries from `subdir` (line 130: `scan_root = self._validate_path(path) if path else self._root_path`). But the stale check at line 206 queries *all* entries for the connector: `WHERE connector_id = $1::uuid`. So entries outside `subdir` that were previously indexed appear stale and get deleted. If someone indexes a subdirectory, they lose their index for everything else.

The fix is simple: skip stale cleanup when `path` is not `None`, or scope the stale query to the sub-path prefix.

**Medium — `connector_index` has no retry count column, so failed embeddings retry indefinitely.**

The spec defines `retry_count < 5` for artifact embeddings (`_sweep_pending` enforces this at `embedding.py:212`). But `connector_index` has no `retry_count` column — the table definition at `005_connectors.sql:17-29` and the sweep at `embedding.py:258-269` have no retry limit. If an embedding fails permanently (e.g., content that consistently triggers an API error), `sweep_connector_index` will retry it every 60 seconds forever. This creates unnecessary API traffic and log noise. Not urgent at current scale but worth noting.

**Low — `items_deleted` not surfaced in MCP response.**

`IndexReport` has an `items_deleted` field (added for stale cleanup), but `ConnectorIndexResponse` at `connector_models.py:42-51` doesn't include it, and `server.py:2779-2785` doesn't forward it. Callers can't tell whether stale entries were cleaned up. The spec's `connector_index` response definition doesn't mention deletion counts either, but the information is useful.

**Low — `tool_connector_register` swallows validation failure details.**

At `server.py:2718-2726`, when validation fails, the response returns `status: "error"` but includes no error detail. The `ConnectorRegisterResponse` model has no field for an error message. The actual error is logged but not returned to the caller. This means MCP clients see `status: "error"` with no explanation. Low severity since the response still communicates failure, just not why.

**Low — No `embedding IS NOT NULL` guard in filesystem query SQL.**

At `filesystem_connector.py:281`, the query filters on `embedding_status = 'complete'` but not `embedding IS NOT NULL`. The spec (Section 2.7) calls for this guard. Carried over from the first review. Would only matter with data corruption. Defense-in-depth gap.
